🧠 Efficiency for Tokens

  • Create the plan with a smarter, more expensive agent such as Opus 4.6 as orchestrator.
  • Use long and detailed prompts when needed — even with 1 day of preparation — to reduce rework later.
  • Apply agentic or fleet mode with lower-cost models for execution after the orchestration step.
  • Use a skills folder so you can enable or disable what the agent uses.
  • Cached input can reduce cost dramatically — around 10 times lower cost for reused context.
  • The grill-me skill can ask the user at each step to choose among several options while also suggesting the preferred one.
  • The Jira skill can document in the Jira ticket all interaction with the genAI agent.

💡 Bad Prompts vs Good Prompts

❌ Bad Prompt "write me some code that detects a checkerboard in an image, implement the error handling too"

→ Generic output, no project context, needs many follow-ups, ignores existing work
✅ Good Prompt /plan Previously, I have implemented a simple calibration for the Honda 6V project. The calibration target for this project (for each camera) consists of two april tags, with a checkerboard in the middle. For now, I have a simple implementation which just detects the April tags and does the estimation based on them. I will want to also implement the detection of the checkerboard, to obtain a better estimation. Use /eol-h6v-development for more details about this project. I was thinking on using the tag detection to compute a homography from the target board plane to the image, and use this information to search for the checkerboard crosses (the points where two black squares touch). Regarding how exactly to find the corners. I previously did an analysis on this and the conclusions are: 1. It would be better to use the homography directly to search in the input image, and not do any warping. 2. The best approach for the detection of the crosses would be the "radon cross detector" which uses the localized radon transform for the detection. You can find my report here: @/workspace/entrytrim/my_local_stuff/h6v_checkerboard_detection/poc_experiments/experiments/11_filter_response_comparison/report/checkerboard_cross_detection_experiment_report.tex And a basic python implementation of the radon detector in @/workspace/entrytrim/my_local_stuff/h6v_checkerboard_detection/poc_experiments/experiments/11_filter_response_comparison/run.py (note that for now I am only interested in the basic implementation, with the 4 lines passing through each point, and not the dense implementation) So, I will want you to implement two separate things and then put them together: 1. I will want a library for the radon cross detector. - This will have to be implemented as a separate library in @/workspace/customers/row/honda/honda_elite_b/basic_viper_6v/components/radon_cross_detector - I will want it to have two different implementation: i. The naive implementation which does it similarly to how it is done in the Python script. I don't know if it's better to have a convolution (maybe the compiler will be able to optimize entries which are always zero? or maybe there is the possibility of using some sparse kernels in the code? no idea) or if it is better to have multiple for loops for traversing the lines. Please investigate this during planning. But the idea is that this does not use additional memory, but needs increased runtime. ii. An approach similar to how the integral image is used to compute the response for Haar filters faster. I want a new buffer that for each new element will store the sums up to that element on: the horizontal line, the vertical line and the two diagonals. This will allow us to compute efficiently the integrals passing through that line. It would be nice to use bit fields for this, if it is not too difficult. - Make this templatized on pixel size, so we can use it in the future for different projects with 8 bits per pixel 16 bits per pixel and so on. - Use TImageViews for inputs, do not couple the API to concrete image dimensions. - Implement unit tests for this - In the future we'll also implement an algorithm to find the maximum response with subpixel precision, but for now this is not in the scope of this task. 2. I will want a library which computes homographies. - This will have to be implemented as a separate library in @/workspace/customers/row/honda/honda_elite_b/basic_viper_6v/components/eol_homography/ - As a reference, you can use the code in @/workspace/entrytrim/components/viper/c3d/april_tag_detector/src/viper/april_tag_detector/april_tag_detector_compute_homography.hpp which implements the DLT algorithm (for now, just copy-paste what's needed). - The current implementation is hardcoded to work only for a correspondence of exactly 4 points. You will have to extend it to work with more than just 4 point correspondences (for this Honda 6V usecase we have 8 point correspondences given by the corners of the april tags) - There is an implementation for singular value decomposition (SVD) which you might find useful here: @/workspace/entrytrim/components/common/algebra/include/algebra/svd_factorization.hpp - Also, please implement unit testing for this (you can use the tests in the existing implementation as a starting point) 3. Put everything together inside @/workspace/customers/row/honda/honda_elite_b/basic_viper_6v/components/interface_honda_6v/src/eol_calibration_process.hpp such that after the tags are detected, we compute the homography from the board to the image and then for each cross on the checkerboard target, we search for the cross in a smallish neighborhood around the expected position (given the homography) and select the cross there if we find any. Then, all the correspondences should be passed to the estimator to obtain the calibration. I do not exactly expect unit tests for this part. But you should ensure that @/workspace/customers/row/honda/honda_elite_b/basic_viper_6v/checkbuilds.sh still works and also that @/workspace/customers/row/honda/honda_elite_b/basic_viper_6v/apps/main_eol.cpp still works (you might need to edit the TagAndCheckerboardSpec with the proper values for the checkerboard target, I think they are just stubbed for now) Some personal preferences regarding the implementation: - if it makes sense, try to use the pattern where you return xcf::expected and implement nice enums with possible failure cases - prefer using xcf::wide_contract unless it makes sense to use narrow_contract (and ensure you do the necessary checks beforehand!) - use range-for loops, possibly with xcf::zip and xcf::enumerate if possible, instead of old style loops. Please try to make the plan with enough details such that it will be able to be implemented by a fleet of smaller models (like Sonnet and Haiku). Also use the skill /grill-me to ensure that we are aligned on the plan before starting the implementation.
